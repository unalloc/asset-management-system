### 1.1 Requirement Description

Manage enterprise hierarchical structure: group → business unit → factory → department multi-level organizational structure. This requirement establishes the foundational multi-company data hierarchy that underpins all subsequent module 6 requirements. The organizational hierarchy defines the data scope levels, access control boundaries, and data isolation requirements for the entire asset management system.

## 3. Business Rules Enhancement

### 3.1 Pre-conditions

| Pre-condition ID | Description | Validation Logic | Error Code | Error Message |
|------------------|-------------|------------------|------------|---------------|
| PRE-ORG-001 | User has organization management permission | User.Role IN ('SystemAdmin', 'OrganizationAdmin') OR User.Permissions CONTAINS 'ORGANIZATION_MANAGE' | ORG-401 | "User does not have organization management permission" |
| PRE-ORG-002 | Parent organization exists (for non-root nodes) | IF Input.ParentOrgID IS NOT NULL THEN EXISTS(SELECT 1 FROM Organization WHERE OrganizationID = Input.ParentOrgID AND Status = 'Active') | ORG-402 | "Parent organization does not exist or is inactive" |
| PRE-ORG-003 | Organization types are predefined | Input.OrganizationType IN ('Group', 'Company', 'Division', 'Plant', 'Department', 'WorkCenter', 'VirtualOrg') | ORG-403 | "Invalid organization type. Valid types: Group, Company, Division, Plant, Department, WorkCenter, VirtualOrg" |
| PRE-ORG-004 | Organization code format is valid | Input.OrgCode matches OrganizationCodeRegex for parent org type | ORG-404 | "Organization code format does not comply with coding rules" |
| PRE-ORG-005 | User has write permission at parent org level | User.AccessLevel >= ParentOrg.Level + 1 | ORG-405 | "User does not have permission to create organization under parent" |

**Pre-condition Validation Matrix:**

```
VALIDATION_ORDER:
  1. PRE-ORG-003 (Type validation - fastest fail)
  2. PRE-ORG-001 (Permission check)
  3. PRE-ORG-002 (Parent exists - if applicable)
  4. PRE-ORG-004 (Code format)
  5. PRE-ORG-005 (Access control)

IF validation FAILS at any step, RETURN error immediately without proceeding.
```

### 3.2 Post-conditions

| Post-condition ID | Description | Success Criteria | Verification Method |
|-------------------|-------------|-------------------|---------------------|
| POST-ORG-001 | Organization node created | OrganizationID IS NOT NULL AND Organization row persisted | SELECT after INSERT verification |
| POST-ORG-002 | Hierarchy relationships established | ParentOrgID reference valid, MaterializedPath updated, Level = Parent.Level + 1 | Hierarchy query verification |
| POST-ORG-003 | Audit trail recorded | AuditLog entry created with Action='CREATE', OldValue IS NULL, NewValue = full organization snapshot | Audit log query |
| POST-ORG-004 | Child count updated on parent | Parent.ChildCount = Parent.ChildCount + 1 | Parent org query after creation |

**Post-condition Verification:**

After successful organization creation:
1. Query Organization table for new OrganizationID
2. Verify MaterializedPath matches pattern: Parent.MaterializedPath + OrganizationID + '/'
3. Verify Level = Parent.Level + 1 (or 0 if root)
4. Confirm AuditLog entry exists with CREATE action

**Calculation Logic (if applicable):**

```
-- Materialized Path Calculation
NEW_PATH = CONCAT(IFNULL(Parent.MaterializedPath, ''), NEW_ORG_ID, '/')

-- Level Calculation

-- Child Count Update
UPDATE Organization SET ChildCount = ChildCount + 1 WHERE OrganizationID = ParentOrgID
```

### 5.3 Data Isolation

| Isolation Requirement | Implementation | Verification Method |
|----------------------|----------------|-------------------|
| Company A cannot view Company B organizations | MaterializedPath + OrganizationType filtering | Unit test: cross-company query returns empty |
| Legal entity data isolated for SOX | OrganizationType='Company' is audit boundary | SOX audit query verification |
| Plant-level data isolated for OSHA reporting | Plant-level filtering on safety records | OSHA report generation test |
| GDPR data residency enforced | JurisdictionCode attribute + data residency rules | Compliance test per jurisdiction |

**Data Segregation Strategy:**

```
-- Database Level (PostgreSQL/Oracle)
- Organization table includes CompanyCode as partition key
- Row-level security (RLS) policies enforce organization-based filtering
- Example: CREATE POLICY org_isolation_policy ON organization
           USING (current_user_org_id IN (SELECT organization_id FROM org_path_function(authorized_orgs)))

-- Application Level
- Session context contains user's authorized_organization_ids array
- All queries filter by authorized_organization_ids
- Organization service throws ORG-408 if user attempts cross-scope access

-- API Level
- X-Organization-ID header required for all organization-scoped operations
- API gateway validates organization access before forwarding request
```
