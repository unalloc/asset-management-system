# 详细需求规格 - 资产利用与分配管理 (2.7)

## 概述
本模块专注于优化资产利用效率，包括资产分配计划、使用请求审批、预留系统、实时可用状态跟踪、利用率计算、成本分配、设备调度、共享池管理等，旨在最大化资产价值同时控制使用成本。

## 功能详细规格

### 2.7.1 资产分配计划
**功能ID**: F-2.7.1  
**功能名称**: 年度/季度资产分配计划  
**功能描述**: 支持制定年度和季度资产分配计划，基于业务需求预测、预算限制和资产可用性，优化资产配置。

**业务规则**:
1. 分配计划周期：年度计划（财政年度）、季度计划（业务季度）
2. 基于业务需求预测：部门预算、项目计划、生产计划
3. 考虑资产可用性：现有资产、计划采购资产、即将处置资产
4. 优化目标：最大化利用率、最小化闲置、平衡工作负载
5. 计划审批：多级审批（部门→财务→管理层）
6. 计划调整：支持计划期内调整，记录变更原因
7. 计划与实际对比：跟踪计划执行情况，分析偏差
8. 历史参考：参考历史分配数据和利用率

**前置条件**:
- 业务需求预测数据可用
- 资产清单和可用性数据完整
- 预算周期已定义
- 审批流程已配置

**后置条件**:
- 资产分配计划生成
- 审批记录保存
- 分配任务创建
- 监控指标设定

**输入数据**:
1. 计划周期（年度/季度）
2. 业务需求预测（部门、项目、产能）
3. 可用资产清单（现有、计划采购）
4. 预算限制
5. 优化目标权重
6. 审批人列表
7. 历史分配参考数据
8. 特殊要求（合规、安全、技术）

**输出数据**:
1. 资产分配计划文档
2. 分配建议矩阵（资产→部门/项目）
3. 利用率预测
4. 预算需求
5. 审批状态
6. 计划执行跟踪仪表板

**验证规则**:
1. 计划周期必须与财务周期对齐
2. 需求预测必须基于可靠数据源
3. 资产可用性必须考虑维护计划
4. 预算不能超支（警告阈值）
5. 优化目标必须可量化

**错误处理**:
1. 数据不足：标记计划为"初步"，需要更多数据
2. 资源冲突：提示冲突并建议替代方案
3. 审批人无效：提示选择有效审批人
4. 优化无解：放宽约束或人工调整

**优先级**: 高  
**相关用户故事**: US-2.7.1, US-2.7.2

### 2.7.2 使用请求和审批工作流
**功能ID**: F-2.7.2  
**功能名称**: 资产使用请求审批  
**功能描述**: 管理资产使用请求的提交、多级审批、条件批准、拒绝和反馈的完整工作流。

**业务规则**:
1. 请求类型：单次使用、重复使用、长期借用、紧急使用
2. 审批级别：基于资产价值、敏感度、使用时长分级审批
3. 并行审批：多个部门同时审批（技术、安全、财务）
4. 条件批准：附带条件的批准（培训完成、安全检查）
5. 自动批准：低风险、低价值请求自动批准
6. 审批超时：自动升级或提醒
7. 审批历史：完整记录审批过程、意见、附件
8. 移动审批：支持移动端审批操作

**前置条件**:
- 资产可分配（非维修、非预留、非处置中）
- 请求人有有效账户和权限
- 审批规则已定义
- 审批人可用

**后置条件**:
- 请求状态更新（批准、拒绝、需修改）
- 审批记录保存
- 通知发送给相关人员
- 资产状态更新（如预留）

**输入数据**:
1. 请求人信息
2. 资产ID列表
3. 使用开始和结束时间
4. 使用目的
5. 使用地点
6. 特殊要求
7. 审批规则引用
8. 附件（业务理由、风险评估）

**输出数据**:
1. 使用请求单号
2. 审批任务分配
3. 审批状态跟踪
4. 审批完成通知
5. 审批历史报告
6. 超时升级记录

**验证规则**:
1. 使用时间必须在资产可用时间内
2. 请求人必须有相应权限
3. 审批规则必须覆盖请求场景
4. 必填信息完整
5. 附件格式和大小符合要求

**错误处理**:
1. 资产不可用：提示替代资产或建议其他时间
2. 权限不足：拒绝请求并通知管理员
3. 审批规则缺失：使用默认规则并提示配置
4. 时间冲突：检测冲突并提示调整

**优先级**: 高  
**相关用户故事**: US-2.7.3, US-2.7.4

### 2.7.3 资产预留系统
**功能ID**: F-2.7.3  
**功能名称**: 资产预留与冲突检测  
**功能描述**: 提供资产预留功能，支持提前预约资产使用，自动检测和解决预留冲突。

**业务规则**:
1. 预留类型：硬预留（保证可用）、软预留（可被更高优先级覆盖）
2. 冲突检测：实时检测时间、地点、资产冲突
3. 冲突解决：优先级规则（紧急>常规，高管>员工，生产>行政）
4. 预留修改：允许修改预留时间、资产、地点
5. 预留取消：提前取消政策，违约记录
6. 预留提醒：预留开始前提醒（邮件、消息）
7. 预留历史：完整预留记录，用于分析和优化
8. 外部系统集成：与会议室预订、生产计划系统集成

**前置条件**:
- 资产可预留（非永久分配、非维修）
- 请求人有权预留
- 预留规则已定义
- 冲突检测算法已配置

**后置条件**:
- 资产预留状态更新
- 预留记录创建
- 冲突解决（如发生）
- 通知发送

**输入数据**:
1. 预留人信息
2. 资产ID或资产类型
3. 预留开始和结束时间
4. 预留地点
5. 预留优先级
6. 预留类型（硬/软）
7. 替代选项偏好
8. 冲突解决偏好

**输出数据**:
1. 预留确认单
2. 冲突检测报告
3. 预留日历视图
4. 提醒设置
5. 预留修改历史
6. 违约记录（如适用）

**验证规则**:
1. 预留时间必须合理（不超过最大预留时长）
2. 预留人必须有预留权限
3. 优先级必须符合公司政策
4. 预留修改必须在合理时间内（如提前24小时）
5. 冲突解决方案必须可接受

**错误处理**:
1. 冲突无法自动解决：人工干预流程
2. 预留超限：提示已达到最大预留限制
3. 规则冲突：提示规则冲突需要管理员调整
4. 系统集成失败：降级到手动模式并记录

**优先级**: 高  
**相关用户故事**: US-2.7.5, US-2.7.6

### 2.7.4 实时资产可用状态跟踪
**功能ID**: F-2.7.4  
**功能名称**: 实时可用状态监控  
**功能描述**: 提供实时资产可用状态跟踪，包括当前状态、位置、使用人、预计可用时间，支持动态更新和通知。

**业务规则**:
1. 状态类型：可用、使用中、维修中、预留、停用、处置中
2. 实时更新：状态变更立即反映在所有界面
3. 位置跟踪：资产当前位置（GPS、RFID、手动更新）
4. 使用人跟踪：当前使用人/责任人信息
5. 预计可用时间：基于当前使用结束时间或维修完成时间
6. 状态历史：状态变更历史，用于分析和审计
7. 自动状态转换：基于规则自动更新状态（如维修完成→可用）
8. 多维度视图：按状态、位置、类型等筛选和分组

**前置条件**:
- 资产状态跟踪已启用
- 状态更新机制就绪（手动或自动）
- 用户有权限查看状态
- 实时通信基础设施可用

**后置条件**:
- 状态信息最新准确
- 相关方收到状态变更通知（如需要）
- 历史记录保存
- 仪表板数据更新

**输入数据**:
1. 资产ID
2. 新状态
3. 状态变更时间
4. 变更原因
5. 位置信息（如变更）
6. 使用人信息（如变更）
7. 预计可用时间
8. 变更人/系统

**输出数据**:
1. 实时状态显示
2. 状态变更通知
3. 状态历史报告
4. 可用资产列表
5. 预计可用时间表
6. 异常状态警报

**验证规则**:
1. 状态变更必须合理（如不能从"处置中"变为"可用"）
2. 位置信息必须有效（如存在的地点）
3. 使用人必须为有效员工
4. 预计时间必须基于可靠数据
5. 变更人必须有相应权限

**错误处理**:
1. 无效状态转换：拒绝变更并提示允许的转换
2. 位置无效：提示选择有效位置
3. 权限不足：拒绝变更并通知管理员
4. 数据不一致：触发数据验证流程

**优先级**: 高  
**相关用户故事**: US-2.7.7, US-2.7.8

- 续约提醒设置

**输入数据**:

4. 合同条款（期限、费用、SLA）

7. 费用计算规则

4. SLA跟踪仪表板

3. 费用计算错误：财务部门审核调整

### 2.7.6 资产利用率计算
**功能ID**: F-2.7.6  
**功能名称**: 利用率计算与分析  
**功能描述**: 计算和分析资产利用率，包括时间利用率、产能利用率、价值利用率，支持多种计算方法和基准比较。

**业务规则**:
1. 利用率类型：时间利用率（实际使用时间/可用时间）、产能利用率（实际产出/设计产能）、价值利用率（产生价值/资产价值）
2. 计算方法：基于传感器数据、人工报告、系统日志
3. 基准设定：行业基准、历史基准、目标基准
4. 分析维度：按资产、部门、时间段、资产类型
5. 趋势分析：利用率变化趋势、季节性模式
6. 根本原因分析：低利用率原因（维护过多、需求不足、配置问题）
7. 优化建议：基于分析结果提出优化建议
8. 报告生成：定期利用率报告、异常警报

**前置条件**:
- 使用数据可获取（传感器、日志、人工输入）
- 可用时间定义明确（考虑班次、维护窗口）
- 基准数据可用
- 分析模型已配置

**后置条件**:
- 利用率指标计算完成
- 分析报告生成
- 优化建议提出
- 绩效指标更新

**输入数据**:
1. 资产ID
2. 使用数据（开始时间、结束时间、产出、价值）
3. 可用时间定义
4. 基准数据
5. 分析参数
6. 报告周期
7. 警报阈值
8. 优化目标

**输出数据**:
1. 利用率指标
2. 分析报告
3. 趋势图表
4. 根本原因分析
5. 优化建议
6. 异常警报
7. 绩效对标结果

**验证规则**:
1. 使用数据必须准确可靠
2. 可用时间定义必须合理
3. 基准数据必须可比
4. 计算方法必须一致
5. 分析参数必须在合理范围内

**错误处理**:
1. 数据缺失：使用估算值并标记不确定性
2. 定义冲突：人工评审确定正确定义
3. 计算错误：重新计算并验证逻辑
4. 基准不可比：调整基准或使用替代基准

**优先级**: 高  
**相关用户故事**: US-2.7.11, US-2.7.12

### 2.7.7 设备效率分析
**功能ID**: F-2.7.7  
**功能名称**: 设备综合效率分析  
**功能描述**: 计算和分析制造设备的综合效率（OEE）、产能利用率、设备效能，支持生产绩效改进。

**业务规则**:
1. OEE计算：可用率 × 性能率 × 质量率
2. 数据来源：与MES、SCADA、PLC系统集成获取实时数据
3. 损失分类：计划停机、非计划停机、速度损失、质量损失
4. 根本原因分析：识别效率损失的根本原因（设备、工艺、人员、材料）
5. 改进跟踪：改进措施实施后效率变化跟踪
6. 对标分析：设备间、工厂间、行业对标
7. 实时监控：OEE实时仪表板、异常警报
8. 报告生成：班次报告、日报告、周报告、月报告

**前置条件**:
- 设备数据可获取（MES/SCADA集成）
- OEE计算参数已配置（计划时间、理论周期时间）
- 损失分类定义明确
- 分析工具可用

**后置条件**:
- OEE指标计算完成
- 损失分析报告生成
- 改进建议提出
- 绩效数据更新

**输入数据**:
1. 设备ID
2. 生产数据（运行时间、停机时间、产出数量、废品数量）
3. 计划时间
4. 理论周期时间
5. 损失分类规则
6. 分析时间段
7. 对标数据
8. 改进措施记录

**输出数据**:
1. OEE指标（可用率、性能率、质量率、综合OEE）
2. 损失分析报告
3. 根本原因分析
4. 改进建议
5. 实时OEE仪表板
6. 对标分析结果
7. 趋势图表

**验证规则**:
1. 生产数据必须准确及时
2. 计划时间必须合理（考虑班次、维护）
3. 理论周期时间必须基于工程标准
4. 损失分类必须一致
5. 计算逻辑必须符合OEE标准

**错误处理**:
1. 数据集成失败：使用最后已知数据并标记
2. 参数缺失：使用默认参数并提示配置
3. 计算异常：检查数据异常并重新计算
4. 对标数据不可用：跳过对标分析

**优先级**: 高  
**相关用户故事**: US-2.7.13, US-2.7.14

### 2.7.8 闲置资产识别
**功能ID**: F-2.7.8  
**功能名称**: 闲置资产识别与优化  
**功能描述**: 识别长期闲置或低利用率资产，分析闲置原因，提出优化方案（重新部署、共享、出售、处置）。

**业务规则**:
1. 闲置定义：基于时间阈值（如30天未使用）、利用率阈值（如<5%）
2. 自动识别：定期扫描资产使用数据，识别闲置资产
3. 原因分析：闲置原因分类（需求变化、技术过时、配置问题、位置不当）
4. 优化方案：重新部署到有需求部门、加入共享池、出售、处置、出租
5. 成本分析：闲置成本（折旧、维护、空间占用）vs 优化方案成本
6. 工作流：闲置识别→原因分析→方案建议→审批→执行
7. 跟踪验证：优化方案执行后效果跟踪
8. 预防措施：基于闲置分析改进采购和分配决策

**前置条件**:
- 资产使用数据完整
- 闲置阈值定义明确
- 优化方案模板可用
- 审批流程定义

**后置条件**:
- 闲置资产列表生成
- 优化方案建议提出
- 审批工作流启动（如需要）

**业务规则**:

2. 平衡目标：均衡利用率、最小化等待时间、最小化总成本
3. 平衡方法：资产重新分配、工作重新分配、共享池扩大
4. 约束考虑：资产移动成本、技术兼容性、地理位置
5. 平衡时机：定期平衡（季度）、触发平衡（严重失衡）
6. 平衡方案：自动生成平衡方案，人工审批执行
7. 平衡效果跟踪：平衡前后绩效对比
8. 预防措施：基于负载预测的预防性平衡

**前置条件**:
- 各部门/地点负载数据可用
- 平衡算法配置
- 约束条件定义
- 审批流程定义

**后置条件**:
- 负载分析完成
- 平衡方案生成
- 审批工作流启动（如需要）
- 平衡执行计划制定

**输入数据**:
1. 部门/地点负载数据
2. 平衡目标权重
3. 平衡方法参数
4. 约束条件（移动成本、兼容性、地理）
5. 平衡触发条件
6. 审批人列表
7. 跟踪指标
8. 预防规则

**输出数据**:
1. 负载分析报告
2. 平衡方案建议
3. 成本效益分析
4. 审批状态
5. 平衡执行计划
6. 效果跟踪仪表板
7. 预防建议

**验证规则**:
1. 负载数据必须准确可比
2. 平衡目标必须可量化
3. 约束条件必须全面
4. 平衡方案必须可行
5. 成本效益必须正净收益

**错误处理**:
1. 数据不可比：标准化数据或使用相对指标
2. 无可行方案：放松约束或调整目标
3. 审批拒绝：记录原因并调整方案
4. 执行困难：调整执行计划或分阶段执行

**优先级**: 中  
**相关用户故事**: US-2.7.31, US-2.7.32

### 2.7.17 关键资产紧急分配
**功能ID**: F-2.7.17  
**功能名称**: 紧急分配流程  
**功能描述**: 为关键资产提供紧急分配流程，支持快速响应突发事件（生产中断、安全事故、紧急维修），最小化业务影响。

**业务规则**:
1. 关键资产定义：对业务连续性和安全至关重要的资产
2. 紧急场景：生产中断、安全事故、自然灾害、紧急维修、关键任务支持
3. 快速通道：简化审批流程，预设紧急审批人，自动优先级最高
4. 冲突解决：紧急请求可中断非紧急使用，有补偿机制
5. 通知机制：紧急分配自动通知相关方（安全、管理、受影响用户）
6. 事后处理：紧急使用后评估影响，补办正式手续，分析改进
7. 演练测试：定期紧急分配演练，确保流程有效
8. 审计跟踪：完整记录紧急分配原因、过程、结果

**前置条件**:
- 紧急场景发生或预警
- 关键资产定义明确
- 紧急审批人可用
- 通知系统就绪

**后置条件**:
- 关键资产紧急分配完成
- 业务影响最小化
- 事后处理启动
- 审计记录保存

**输入数据**:
1. 紧急场景类型
2. 关键资产需求
3. 紧急程度级别
4. 影响范围估计
5. 紧急审批人联系
6. 受影响用户列表
7. 事后处理规则
8. 审计要求

**输出数据**:
1. 紧急分配记录
2. 快速审批结果
3. 自动通知记录
4. 冲突解决记录
5. 事后评估报告
6. 改进建议
7. 审计跟踪报告
8. 演练测试结果

**验证规则**:
1. 紧急场景必须真实有效（防滥用）
2. 关键资产必须确实需要
3. 紧急程度必须合理
4. 审批人必须有紧急授权
5. 事后处理必须完整

**错误处理**:
1. 滥用检测：疑似滥用触发调查
2. 审批人不可用：升级到备用审批人或自动批准
3. 资产不可用：寻找替代资产或启动应急计划
4. 流程失效：降级到手动应急流程并事后改进

**优先级**: 高  
**相关用户故事**: US-2.7.33, US-2.7.34

### 2.7.18 使用合规监控
**功能ID**: F-2.7.18  
**功能名称**: 使用合规监控  
**功能描述**: 监控资产使用合规性，包括政策遵守、安全规程、授权使用、数据安全等，支持违规检测和纠正。

**业务规则**:
1. 合规策略：使用政策、安全规程、授权矩阵、数据安全要求
2. 监控方法：自动检查（系统日志）、人工报告、定期审计
3. 违规检测：未授权使用、超范围使用、不安全操作、数据泄露风险
4. 纠正措施：违规通知、使用暂停、培训要求、纪律处分
5. 合规报告：合规状态报告、违规趋势分析、整改跟踪
6. 预防措施：合规培训、政策宣传、系统控制（预防违规）
7. 集成监控：与安全系统（SIEM）、访问控制系统集成
8. 法规适配：支持不同地区法规要求（GDPR、数据安全法）

**前置条件**:
- 合规策略已定义和发布
- 监控机制就绪（系统、人员）
- 违规处理流程定义
- 报告模板可用

**后置条件**:
- 合规监控持续运行
- 违规检测和处理
- 合规报告生成
- 整改措施跟踪

**输入数据**:
1. 合规策略定义
2. 使用数据（日志、报告）
3. 监控规则
4. 违规检测阈值
5. 纠正措施流程
6. 报告周期和要求
7. 集成接口参数
8. 法规要求库

**输出数据**:
1. 合规监控报告
2. 违规检测警报
3. 纠正措施记录
4. 整改跟踪状态
5. 趋势分析
6. 预防措施建议
7. 法规合规证明
8. 审计准备材料

**验证规则**:
1. 合规策略必须合法合理
2. 监控方法必须有效覆盖
3. 违规检测必须准确（低误报）
4. 纠正措施必须适当
5. 报告必须满足监管要求

**错误处理**:
1. 策略冲突：识别并解决策略冲突
2. 监控盲点：定期评估和增强监控
3. 误报处理：优化检测规则，减少误报
4. 违规处理争议：建立申诉和仲裁机制

**优先级**: 高  
**相关用户故事**: US-2.7.35, US-2.7.36
