### 1.1 Requirement Description

This requirement defines the comprehensive multi-company data hierarchy structure and role-based access control framework for the entire asset management system. It establishes the authoritative organizational structure, data scope levels, access control matrices, data isolation rules, consolidation mechanisms, and data sovereignty requirements that all other requirements must conform to.

This requirement serves as the **foundational architecture** for:
- All data scoping and visibility rules
- All role-based access control decisions
- All inter-company data flow controls
- All regulatory compliance (GDPR, SOX, IFRS) at the data layer

### 5.4 Access Control Implementation Rules

**Rule AC-001: Hierarchical Access**
- Users may only access data at or below their assigned organizational level
- Exception: Users with explicit cross-organizational grants may access granted levels
- Exception: Users with GROUP-level roles may access all levels

**Rule AC-002: Downward Inheritance**
- Permissions granted at a higher level automatically apply to all child levels
- A Company Admin can access all Divisions, Plants, Departments, and Cost Centers within their Company

**Rule AC-003: Upward Restriction**
- Permissions at a lower level do not grant access to parent levels
- A Department Admin cannot access Division, Company, or Group data

**Rule AC-004: Scope Boundary**
- Cross-company access is blocked by default unless explicit grants exist
- Cross-company grants require Company Admin approval at both companies

**Rule AC-005: Role Exclusivity**
- A user may hold only one operational role per organizational assignment
- A user may hold multiple viewer roles across organizations
- System Integration accounts are excluded from this rule

**Rule AC-006: Role Constraints**
- Senior Technician may approve work orders up to $10,000
- Plant Admin may dispose assets up to $50,000
- Company Admin may dispose assets up to $500,000
- Assets exceeding limits require higher-level approval

## 6. Data Isolation Rules

### 6.1 Row-Level Security Implementation

**Database-Level RLS:**

```sql
-- Enable Row-Level Security
ALTER TABLE assets ENABLE ROW LEVEL SECURITY;

-- RLS Policy: Users see only their organizational scope
CREATE POLICY asset_visibility_policy ON assets
    USING (
        -- Group Admin sees all
        current_user_role() = 'GROUP_ADMIN'
        OR
        -- Company Admin sees company and children
        (
            current_user_role() IN ('COMPANY_ADMIN', 'DIVISION_ADMIN', 'PLANT_ADMIN', 'DEPARTMENT_ADMIN')
            AND company_code IN (
                SELECT child_code 
                FROM organization_hierarchy 
                WHERE parent_code = current_user_company_code()
                OR code = current_user_company_code()
            )
        )
        OR
        -- Technicians see assigned scope
        (
            current_user_role() = 'TECHNICIAN'
            AND plant_code = current_user_plant_code()
        )
    );
```

### 6.2 API-Level Filtering Requirements

**API Request Processing:**

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         API Request Flow                                │
└─────────────────────────────────────────────────────────────────────────┘

1. Request Received
         │
         ▼
2. Extract JWT Claims
   - user_id
   - organization_codes[]  (array of accessible org units)
   - roles[]
   - company_code
         │
         ▼
3. Determine Data Scope
   - Map roles to permitted organization levels
   - Apply hierarchy inheritance rules
   - Check for explicit cross-org grants
         │
         ▼
4. Inject Organization Filters
   - Add WHERE clause: organization_level IN (accessible_orgs)
   - Add JOINs: organization_hierarchy for inheritance
   - Add column projections: filter sensitive fields
         │
         ▼
5. Execute Query
   - RLS policies enforced by database
   - Query rewrite adds additional filters
         │
         ▼
6. Filter Response
   - Remove any data outside scope (defense in depth)
   - Audit log the access
         │
         ▼
7. Return Filtered Response
```

### 6.3 Query Rewriting Requirements

**Automatic Query Augmentation:**

| Input Query | Augmented Query |
|-------------|-----------------|
| `SELECT * FROM assets` | `SELECT * FROM assets WHERE company_code IN (user's accessible companies)` |
| `SELECT * FROM assets WHERE status = 'ACTIVE'` | `SELECT * FROM assets WHERE company_code IN (...) AND status = 'ACTIVE'` |
| `UPDATE assets SET location = 'X' WHERE id = 123` | `UPDATE assets SET location = 'X' WHERE id = 123 AND company_code IN (...)` |
| `DELETE FROM assets WHERE id = 456` | `DELETE FROM assets WHERE id = 456 AND company_code IN (...)` |

**Query Validation Rules:**

| Validation | Rule | Error Code | Error Message |
|------------|------|------------|---------------|
| Cross-company detection | Query must not target non-accessible companies | ISO-001 | "Access denied to specified company" |
| Organization code required | All asset queries must include organization filter | ISO-002 | "Organization scope required" |
| Grant verification | Cross-company access requires valid grant | ISO-003 | "Cross-company grant required" |
| Batch size limit | Max 1000 records per cross-company batch | ISO-004 | "Batch size exceeds limit" |

### 6.4 Cross-Company Data Access Prevention

**Prevention Layers:**

| Layer | Mechanism | Verification |
|-------|-----------|--------------|
| **Database** | RLS policies per table | Unit test: cross-company query returns empty |
| **Application** | Query builder with scope injection | Integration test: API returns only scoped data |
| **API Gateway** | JWT validation with scope claims | Security test: token without scope rejected |
| **UI** | Organization selector with scope filtering | UI test: no cross-company records displayed |

**Blocked Scenarios:**

| Scenario | Prevention | User Feedback |
|----------|------------|----------------|
| Direct SQL to bypass app | Database RLS blocks query | Error logged, security alert |
| JWT token manipulation | Server-side scope recalculation | Token invalidated, error returned |
| API parameter injection | Whitelist validation, scope enforcement | "Invalid organization code" |
| UI manipulation | Server-side filtering, no client-only controls | Empty result set returned |

### 7.3 Consolidation Methods

**Method 1: Full Consolidation**

| Condition | Application | Method |
|-----------|-------------|--------|
| Ownership ≥ 50% | Subsidiary assets, liabilities, income | 100% of assets/liabilities |
| Control exists | Operating entities under company control | Add full amounts, subtract minority |

**Method 2: Proportionate Consolidation**

| Condition | Application | Method |
|-----------|-------------|--------|
| Joint ventures (IFRS 11) | Joint operation interests | Share of assets/liabilities/income |
| 50/50 arrangements | Shared facility operations | Equal proportion of costs/benefits |

**Method 3: Equity Method**

| Condition | Application | Method |
|-----------|-------------|--------|
| Ownership 20-50% | Associate companies | Investment at cost + share of earnings |
| Significant influence | Minority holdings | Recognition of proportional income |

### 9.3 Business Logic - Hierarchy Validation

**Decision Tree:**

```
START: User accesses data
           │
           ▼
Is user authenticated?
    ├─► NO ──► Reject with DH-010
    └─► YES
           │
           ▼
Get user organization context
    - organization_codes[]
    - roles[]
    - effective_date
           │
           ▼
Is user role = SYSTEM_ADMIN or GROUP_ADMIN?
    ├─► YES ──► Bypass scope filtering, full access
    └─► NO
           │
           ▼
For each data record:
    Check IF record.organization_code IN user.organization_codes[]
           │
           ├─► YES ──► Check role permissions at that level
           │           ├─► CREATE ──► User has create permission at level?
           │           ├─► READ   ──► User has read permission at level?
           │           ├─► UPDATE ──► User has update permission at level?
           │           └─► DELETE ──► User has delete permission at level?
           │
           └─► NO ──► Check for explicit cross-org grant
                       ├─► Grant exists AND valid ──► Apply grant permissions
                       └─► No grant ──► Block access, log to audit
```

|--------------|----------|-------------------|-------------------|
| EC-DH-001 | User transfers between organizations | Immediate access change, invalidate session | Use new org context, expire old grants |

## 11. Implementation Requirements

### 11.1 Default Hierarchy Setup

**Initial Setup Sequence:**

```
1. Create Group Organization
   - OrgCode: "GROUP"
   - OrgLevel: 1
   - No Parent
   - Status: ACTIVE

2. Create Company Organizations
   - For each legal entity:
     - OrgCode: Unique code (e.g., "US01", "EU01")
     - OrgLevel: 2
     - Parent: GROUP
     - Status: ACTIVE

3. Create Division Organizations (if applicable)
   - For each business unit:
     - OrgCode: Unique code (e.g., "US01-MFG", "EU01-RETAIL")
     - OrgLevel: 3
     - Parent: Company
     - Status: ACTIVE

4. Create Plant Organizations
   - For each facility:
     - OrgCode: Unique code (e.g., "US01-MFG-CHI", "EU01-RETAIL-PAR")
     - OrgLevel: 4
     - Parent: Division or Company
     - Status: ACTIVE

5. Create Department Organizations (if applicable)
   - For each operational unit:
     - OrgCode: Unique code
     - OrgLevel: 5
     - Parent: Plant
     - Status: ACTIVE

6. Create Cost Centers
   - For each financial accounting unit:
     - OrgCode: Unique code
     - OrgLevel: 6
     - Parent: Department or Plant
     - Status: ACTIVE

7. Assign Default Roles
   - Create SYSTEM_ADMIN at GROUP level
   - Create COMPANY_ADMIN at each Company
   - Create PLANT_ADMIN at each Plant
   - Assign initial admin users

8. Enable Data Isolation
   - Set SystemConfig.DataIsolationEnabled = TRUE
   - Activate RLS policies
   - Configure organization filters
```

### 11.2 Hierarchy Change Workflows

**Organization Creation Workflow:**

| Step | Actor | Action | Validation |
|------|-------|--------|------------|
| 1 | Requester | Submit organization creation request | Required fields: Name, Level, Parent |
| 2 | Parent Org Admin | Approve/Reject creation | Verify parent relationship valid |
| 3 | Security Review | Verify no conflicts with existing | Check for duplicate codes |
| 4 | Compliance Review | Verify regulatory requirements | Country-specific checks |
| 5 | System | Create organization record | Generate unique code |
| 6 | System | Configure default access | Assign parent permissions |
| 7 | System | Enable data scoping | Activate RLS for new org |
| 8 | Notification | Alert relevant users | Parent admin, new org admin |

**Organization Modification Workflow:**

| Change Type | Approval Required | Special Handling |
|-------------|-------------------|------------------|
| Name Change | Parent Admin | Update all references |
| Hierarchy Move | Company Admin + Security | Data migration, access reassignment |
| Status Change (Active→Inactive) | Company Admin | Asset reassignment required |
| Status Change (Inactive→Active) | Company Admin + Compliance | Verify data integrity |
| Merge | Group Admin | Complex data consolidation |
| Split | Group Admin | Division of assets, liabilities |

**Organization Dissolution Workflow:**

```
1. Initiate Dissolution
   - Set status to "PENDING_DISSOLUTION"
   - Lock new asset additions
   - Notify all stakeholders

2. Asset Reassignment
   - Identify all assets in dissolving org
   - For each asset:
     - Select target organization
     - Transfer with audit trail
     - Update location/cost center

3. Data Archival
   - Export all data to archive storage
   - Comply with retention policies
   - Verify archive completeness

4. Access Revocation
   - Disable all user assignments
   - Revoke all cross-org grants
   - Deactivate role permissions

5. Final Validation
   - Verify no orphaned records
   - Confirm zero active users
   - Sign off by compliance

6. Complete Dissolution
   - Set status to "DISSOLVED"
   - Archive organization record
   - Update parent hierarchy
```

### 11.3 Data Migration Rules

| Migration Scenario | Rule | Validation |
|-------------------|------|------------|
| Asset re-org transfer | Maintain full audit trail | Source and target org verified |
| Bulk hierarchy change | Transaction rollback on failure | Atomic operation |
| Historical data preservation | Never delete historical records | Archive before change |
| Cost center reassignment | Re-rate depreciation from move date | Effective date validation |
| Cross-company transfer | Inter-company elimination reset | Both companies approve |
