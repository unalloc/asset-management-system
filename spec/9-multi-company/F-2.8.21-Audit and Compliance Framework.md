### 1.1 Requirement Description

This requirement defines a comprehensive Audit and Compliance Framework that ensures the asset management system meets regulatory compliance across US and EU markets. The framework encompasses audit trail management, SOX compliance controls, GDPR data privacy requirements, ISO 55000 asset management alignment, and regulatory compliance monitoring across all system modules.

### 3.3 Business Logic

**Decision Tree:**

```
START
  │
  ▼
[Data Change Event]
  │
  ├─► Log Event Type (CREATE/UPDATE/DELETE/READ)
  │
  ▼
[Capture Audit Context]
  │
  ├─► User ID
  ├─► Timestamp (UTC)
  ├─► IP Address
  ├─► Session ID
  └─► Reason (if applicable)
  │
  ▼
[Capture Data Changes]
  │
  ├─► Old Value (for UPDATE/DELETE)
  ├─► New Value (for CREATE/UPDATE)
  └─► Changed Fields List
  │
  ▼
[Calculate Hash]
  │
  └─► SHA-256(PreviousHash + EntryData)
  │
  ▼
[Store Audit Entry]
  │
  ▼
[Update Compliance Indicators]
  │
  ▼
END
```

**Business Rules Specification:**

| Rule ID | Rule Name | Condition | Action | Priority |
|---------|-----------|-----------|--------|----------|
| BR-AUDIT-001 | Immutable Audit Log | Any data change | Create audit entry with hash chain | 1 |
| BR-AUDIT-002 | Retention Enforcement | Audit entry age > retention period | Archive to cold storage | 2 |
| BR-AUDIT-003 | Segregation of Duties Check | User attempts privileged action | Verify role permissions against SOD matrix | 1 |
| BR-AUDIT-004 | Personal Data Access Logging | Read of PII fields | Log access with user context | 1 |
| BR-AUDIT-005 | Consent Verification | Processing personal data | Verify valid consent exists | 1 |

### 3.4 Edge Cases

| Edge Case ID | Scenario | Expected Behavior | Handling Approach |
|--------------|----------|-------------------|-------------------|
| EC-AUDIT-001 | Hash chain broken | Detect tampering, alert security | Block access, notify compliance |
| EC-AUDIT-002 | Concurrent modification | Last write wins with all changes logged | Timestamp ordering, full delta capture |
| EC-AUDIT-003 | Audit service unavailable | Queue writes, block critical transactions if required | Circuit breaker, fallback to local logging |
| EC-AUDIT-004 | GDPR erasure request | Anonymize personal data, preserve audit trail | Soft delete with anonymization |
| EC-AUDIT-005 | Mass data export | Log as batch operation, rate limit | Bulk operation tracking |

### 3.5 State Transitions

**Audit Entry State Diagram:**

```
┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│  CREATED    │──────►│  ARCHIVED   │──────►│  DELETED    │
└─────────────┘       └─────────────┘       └─────────────┘
       │                     │                     │
       │                     ▼                     │
       │              ┌─────────────┐              │
       └─────────────►│  EXPORTED   │◄─────────────┘
                      └─────────────┘
```

**Transition Rules:**

| Current State | Event | Next State | Guard Conditions | Side Effects |
|---------------|-------|------------|------------------|---------------|
| CREATED | Retention period expired | ARCHIVED | Age > retention period | Move to cold storage |
| CREATED | Regulatory request | EXPORTED | Valid request, approval | Generate export package |
| ARCHIVED | GDPR erasure request | DELETED | Anonymization complete | Remove PII, keep structural data |
| EXPORTED | Import by external auditor | CREATED | New system entry | New audit domain |

## 5. Audit Trail Framework

### 5.1 Audit Event Types

| Event Type | Description | Fields Captured | Trigger |
|------------|-------------|-----------------|---------|
| CREATE | New record created | All fields, user, timestamp, IP, session | INSERT on any tracked entity |
| UPDATE | Record modified | Old value, new value, changed fields, user, timestamp | UPDATE on any tracked entity |
| DELETE | Record removed | All fields (before deletion), user, timestamp, reason | DELETE on any tracked entity |
| READ | Sensitive data accessed | User, timestamp, data accessed, session | SELECT on sensitive fields |
| LOGIN | User authentication | User, timestamp, IP, success/failure | Authentication event |
| LOGOUT | User session end | User, timestamp, session duration | Session termination |
| EXPORT | Data exported | User, timestamp, data types, record count | Data export operation |
| CONSENT | Consent recorded | Subject, purpose, timestamp, proof | Consent grant/revoke |

### 5.2 Audit Log Structure

**Audit Log Schema:**

```
AuditLog {
    AuditLogID: UUID (PK)
    EntityType: VARCHAR(50)           // Asset, WorkOrder, Depreciation, User, etc.
    EntityID: UUID                   // Reference to the affected record
    Action: ENUM('CREATE', 'UPDATE', 'DELETE', 'READ', 'LOGIN', 'LOGOUT', 'EXPORT', 'CONSENT')
    OldValue: JSON                   // Previous state (null for CREATE)
    NewValue: JSON                   // New state (null for DELETE)
    ChangedFields: ARRAY[VARCHAR]    // List of modified field names
    UserID: UUID (FK)                // User who performed the action
    Username: VARCHAR(100)           // Denormalized for reporting
    UserRole: VARCHAR(50)            // Role at time of action
    Timestamp: DATETIME(UTC)          // When the action occurred
    IPAddress: VARCHAR(45)            // Client IP (IPv4/IPv6)
    SessionID: VARCHAR(100)          // Session identifier
    Reason: TEXT                     // Business reason for change (if applicable)
    CompanyCode: VARCHAR(20)         // Data scope - company
    DivisionCode: VARCHAR(20)        // Data scope - division
    PlantCode: VARCHAR(20)           // Data scope - plant
    DepartmentCode: VARCHAR(20)      // Data scope - department
    Hash: VARCHAR(64)                // SHA-256 hash of this entry
    PreviousHash: VARCHAR(64)        // Hash of previous entry (for chain integrity)
    RequestID: VARCHAR(100)          // Distributed tracing ID
}

AuditLogIndex {
    IndexID: UUID (PK)
    AuditLogID: UUID (FK)
    IndexType: ENUM('BY_ENTITY', 'BY_USER', 'BY_TIMESTAMP', 'BY_ACTION', 'BY_COMPANY')
    IndexKey: VARCHAR(255)           // Indexed value for fast lookup
}
```

### 5.3 Immutable Log Requirements

**Hash Chain Implementation:**

```
For each AuditLog entry:
    EntryData = EntityType + EntityID + Action + OldValue + NewValue + ChangedFields + 
                UserID + Username + Timestamp + IPAddress + SessionID + Reason + CompanyCode
    Hash = SHA256(PreviousHash + EntryData)
    Store Hash and PreviousHash
```

**Immutability Guarantees:**
1. No UPDATE or DELETE operations permitted on AuditLog tables
2. Hash chain provides tamper detection
3. Database-level constraints prevent modification
4. Separate audit access with dual control

### 6.2 Approval Workflows with Audit Trail

**Asset Acquisition Approval Workflow:**

```
1. Requestor creates Asset Acquisition Request
   └─► Audit: CREATE on AssetAcquisitionRequest with Requestor ID

2. Request submitted for approval
   └─► Audit: Action = 'SUBMIT', Status = 'PENDING'

3. Budget Owner reviews and approves/rejects
   └─► Audit: UPDATE on AssetAcquisitionRequest with approval decision
   └─► Audit: Reason field captures approval/rejection rationale

4. If approved: Asset created in system
   └─► Audit: CREATE on Asset with link to AcquisitionRequest

5. If rejected: Request closed with reason
   └─► Audit: UPDATE on AssetAcquisitionRequest with Status = 'REJECTED'
```

**Depreciation Change Approval Workflow:**

```
1. User initiates depreciation parameter change
   └─► Audit: UPDATE on Asset with OldValue/NewValue for changed fields

2. Change enters approval queue
   └─► Audit: Action = 'PENDING_APPROVAL', Approver assigned

3. Finance Approver reviews impact
   └─► Audit: READ on affected depreciation records

4. Approval granted/denied
   └─► Audit: UPDATE with approval decision, Reason required

5. If approved: Depreciation recalculated
   └─► Audit: Depreciation recalculation events logged
```

### 7.1 Personal Data Processing Records (Art. 30)

**Processing Activity Schema:**

```
ProcessingRecord {
    ProcessingRecordID: UUID (PK)
    CompanyCode: VARCHAR(20)
    Purpose: VARCHAR(200)                    // Art. 5(1)(b) - specified, explicit
    LegalBasis: ENUM                          // Art. 6
        - 'CONTRACT'
        - 'LEGAL_OBLIGATION'
        - 'LEGITIMATE_INTEREST'
        - 'CONSENT'
        - 'VITAL_INTEREST'
        - 'PUBLIC_TASK'
    CategoriesOfDataSubjects: ARRAY[VARCHAR] // Employees, Contractors, etc.
    CategoriesOfPersonalData: ARRAY[VARCHAR]  // Name, ID, Location, etc.
    CategoriesOfRecipients: ARRAY[VARCHAR]    // ERP, Payroll, etc.
    ThirdCountryTransfers: BOOLEAN
    TransferSafeguards: JSON                  // SCCs, BCRs, adequacy decision
    RetentionPeriod: VARCHAR(50)
    SecurityMeasures: TEXT
    DPIAReference: VARCHAR(50)
    CreatedAt: DATETIME
    UpdatedAt: DATETIME
}
```

### 7.2 Consent Management

**Consent Record Schema:**

```
ConsentRecord {
    ConsentID: UUID (PK)
    SubjectID: UUID                           // Data subject identifier
    SubjectEmail: VARCHAR(255)                 // For contact purposes
    Purpose: VARCHAR(200)                       // Specific processing purpose
    PurposeDescription: TEXT                   // Detailed explanation
    LegalBasis: 'CONSENT'
    GrantedAt: DATETIME
    ExpiresAt: DATETIME
    RevokedAt: DATETIME
    Proof: TEXT                                // Consent proof (IP, timestamp, etc.)
    WithdrawalMethod: VARCHAR(100)             // How consent can be withdrawn
    CompanyCode: VARCHAR(20)
    IsActive: BOOLEAN
}
```

**Consent Verification Process:**

```
1. Before processing personal data:
   └─► Query ConsentRecord WHERE SubjectID = X AND Purpose = Y AND IsActive = TRUE
   └─► Verify ExpiresAt > CURRENT_TIMESTAMP

2. If valid consent exists:
   └─► Proceed with processing

3. If no valid consent:
   └─► Block processing OR request consent
   └─► Audit: CONSENT_REQUIRED event logged
```

## 13. Enhanced Validation Rules

### 13.1 Field-Level Validation

| Field Name | Data Type | Format | Range | Required | Validation Rule | Error Code | Error Message |
|------------|-----------|--------|-------|----------|-----------------|------------|---------------|
| AuditLog.EntityType | VARCHAR | N/A | Valid entity list | YES | EntityType IN (select list) | VAL-AUDIT-001 | "Invalid entity type for audit logging" |
| AuditLog.Timestamp | DATETIME | ISO 8601 | <= NOW + 1 minute | YES | Timestamp <= CURRENT_TIMESTAMP + 1min | VAL-AUDIT-002 | "Audit timestamp cannot be in the future" |
| AuditLog.Hash | VARCHAR | SHA-256 | 64 characters | YES | Pattern: ^[a-f0-9]{64}$ | VAL-AUDIT-003 | "Invalid hash format" |
| Consent.SubjectID | UUID | UUID v4 | N/A | YES | Valid UUID format | VAL-GDPR-001 | "Invalid subject identifier" |
| Consent.GrantedAt | DATETIME | ISO 8601 | Past dates only | YES | GrantedAt <= CURRENT_TIMESTAMP | VAL-GDPR-002 | "Consent timestamp cannot be in the future" |

### 13.2 Business Rule Validation

| Validation Rule | Description | Condition | Action on Failure | Error Code |
|----------------|-------------|-----------|------------------|------------|
| BRV-AUDIT-001 | Audit log hash chain must be valid | New entry PreviousHash matches last entry Hash | Reject transaction, alert security | VAL-AUDIT-010 |
| BRV-AUDIT-002 | Audit retention cannot be shortened | New retention < existing | Reject change | VAL-AUDIT-011 |
| BRV-GDPR-001 | Active consent required for processing | Processing personal data AND no valid consent | Block processing | VAL-GDPR-010 |
| BRV-GDPR-002 | Erasure must preserve audit trail | Erasure request AND entity has regulatory retention | Anonymize, preserve structural data | VAL-GDPR-011 |
| BRV-SOX-001 | Dual approval for sensitive changes | Asset value change > threshold | Require second approver | VAL-SOX-001 |

| Error Code | Severity | Category | Short Description | Long Description | User Guidance |
|------------|----------|----------|-------------------|------------------|---------------|

| VAL-AUDIT-002 | Error | Audit | Future timestamp | Audit log timestamp cannot be in the future | System time must be synchronized |
| VAL-AUDIT-003 | Error | Audit | Invalid hash | Hash does not match required format | Contact system administrator |
