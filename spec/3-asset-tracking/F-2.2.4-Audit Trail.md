# F-2.2.4 - Audit Trail

## Requirement ID
F-2.2.4

## Function Name
Audit Trail

## Function Description
Record complete audit trails for all asset movements and status changes. All changes must record operator, timestamp, and reason. System must support before/after value comparison. Audit logs must be immutable and support comprehensive query and export capabilities.

## Business Purpose
Maintain a complete, unchangeable history of all asset changes to meet regulatory requirements (SOX, ISO 55001) and support financial auditing, compliance investigations, and operational troubleshooting. The audit trail provides evidence of planned changes and supports performance evaluation and continual improvement.

## Core Business Rules

1. **Immutable Records**: Audit records cannot be modified, updated, or deleted by any user. Only the system can create audit records.

2. **Complete Capture**: Every asset CREATE, UPDATE, DELETE action must be recorded with full context including who made the change, when, and why.

3. **Before/After Comparison**: For updates, capture both the original values and the new values to show exactly what changed.

4. **Integrity Verification**: Each audit record must have a cryptographic integrity marker to detect tampering.

5. **Retention Enforcement**: Audit records must be retained for 7 years minimum (SOX requirement). Records approaching retention limit must be archived to cold storage—not deleted.

6. **Sensitive Data Masking**: When audit logs contain GDPR-protected personal data, appropriate masking must be applied in outputs.

7. **System-Generated Changes**: Even changes initiated by the system must be recorded, marked with "System" as the operator.

8. **Bulk Operations**: When multiple entities are changed in one operation, each entity must have its own separate audit record.

9. **Failed Transactions**: If a transaction fails, partial changes must not be recorded—rollback must occur.

## User Roles and Permissions

| Role | View Own | View Department | View Company | View All | Export |
|------|----------|-----------------|--------------|----------|--------|
| System Admin | Yes | Yes | Yes | Yes | Yes |
| Company Admin | Yes | Yes | Yes | Yes | Yes |
| Division Manager | Yes | Yes | Yes | Yes | Yes |
| Auditor | Yes | Yes | Yes | Yes | Yes |
| Compliance Officer | Yes | Yes | Yes | Yes | Yes |
| Regular User | Own only | No | No | No | No |

**Critical:** No user (including System Admin) can modify or delete audit records. System Admin can only grant VIEW access.

## Data Scope

- **Creates Data:** System only (automatically on any asset change)
- **Views Data:** Company-level users can view their company's audit data; auditors and compliance officers can view all
- **Modifies Data:** NO USER can modify or delete audit records—immutable by design

**Data Segregation:** Company A cannot view Company B's audit data. Audit data is isolated by company code.

## Compliance Notes

**SOX (Sarbanes-Oxley) - Critical:**
- Section 302: Corporate responsibility for financial reports requires audit trail of all asset changes
- Section 404: Internal control assessment requires immutable audit with integrity verification
- Audit trail must support external auditor access
- 7-year retention minimum

**GDPR:**
- Audit logs may contain personal data (operator identity)
- Lawful basis: Legitimate interest for compliance monitoring
- Right to erasure does NOT apply to audit logs (legal obligation exemption under Art. 17(3)(b) and (e))
- Retention: 7 years minimum

**ISO 55001:**
- Clause 7.5.3 requires documented information (records) for asset management system
- Audit trail provides evidence of planned changes and operational logging

**OSHA:**
- 29 CFR 1910 for hazard incident tracking requires 30-year retention for hazmat-related asset changes

**GAMP 5 (Pharma):**
- Computerized systems validation requires audit trail for 21 CFR Part 11 electronic records

## Validation Rules (Plain Language)

1. **Operator Identity**: The operator must be a verified, authenticated user before any audit record is created.

2. **Change Detection**: There must be an actual change to record—no audit entry for unchanged entities.

3. **Timestamp Accuracy**: Timestamps must be within 5 minutes of system time to prevent future-dated or backdated entries.

4. **Hash Integrity**: Each audit record's integrity marker must be verifiable—any mismatch indicates potential tampering.

5. **No Duplicates**: Each audit event can only be recorded once; duplicate detection via idempotency key.

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| System-generated change | Record with "System" as operator identity |
| Bulk operation affecting 100+ assets | Record each asset change as separate audit entry |
| Transaction fails mid-process | No audit record created—rollback all changes |
| Hash collision detected | Append sequence number to ensure uniqueness |
| Audit data approaching 7-year retention | Archive to cold storage, do not delete |
| GDPR right to erasure request | Request denied—legal obligation exemption applies |
| External auditor requests access | Grant time-limited, read-only access |

## Event Types and Retention

| Event Type | Fields Logged | Retention | Access |
|------------|---------------|-----------|--------|
| CREATE | All fields, user, timestamp, reason | 7 years | Auditors only |
| UPDATE | Old value, new value, user, timestamp, reason | 7 years | Auditors only |
| DELETE | Soft delete recorded, hard delete prohibited | 7 years | Auditors only |
| READ (sensitive) | User, timestamp, data accessed | 3 years | Security team only |
| LOGIN/LOGOUT | User, timestamp, IP, session | 7 years | Security team only |
