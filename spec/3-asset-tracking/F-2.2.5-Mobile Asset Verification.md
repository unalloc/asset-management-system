# F-2.2.5 - Mobile Asset Verification

## Requirement ID
F-2.2.5

## Function Name
Mobile Asset Verification

## Function Description
Provide mobile applications for field technicians to verify asset location and condition with offline operation support, barcode/QR code scanning, photo attachments, and real-time synchronization. Verification results update the asset master and trigger exception reporting for discrepancies.

## Business Purpose
Enable field technicians to physically verify assets in remote locations without requiring constant network connectivity. Supports SOX compliance through physical inventory verification, OSHA safety equipment inspection tracking, and GxP compliance for pharmaceutical environments. Delivers real-time visibility into asset condition regardless of location.

## Core Business Rules

1. **Offline-First Operation**: Mobile app must function fully without network connectivity. All verifications queue locally and sync when connection is available.

2. **Asset Identification**: Assets must be verifiable via barcode, QR code, or RFID tag scanning.

3. **Condition Rating**: Each verification must capture a condition rating on a 1-5 scale.

4. **Location Confirmation**: Technician must confirm or update asset location during verification.

5. **Photo Documentation**: Optional photo attachments allowed for condition documentation (compressed if over 5MB).

6. **GPS Capture**: Optional GPS coordinates may be captured to confirm asset location.

7. **Discrepancy Reporting**: Any difference between expected and actual asset state creates an exception record for management review.

8. **Condition Score Auto-Update**: Completion of verification automatically recalculates the asset's overall condition score.

9. **Overdue Alert**: If verification is overdue beyond the grace period, alert is sent to manager.

10. **Duplicate Debouncing**: Ignore duplicate scans of the same asset within a 5-minute window.

11. **Conflict Resolution**: When sync conflicts occur, last-write-wins with conflict logged for review.

## Compliance Notes

**SOX:**
- Physical verification supports Section 404 internal controls
- Periodic inventory counts require verification documentation
- Verification records serve as audit evidence

**OSHA:**
- Safety equipment inspection must be documented per 29 CFR 1910 Subpart I
- Pre-use forklift inspections required and must be recorded

**GAMP 5 (Pharma):**
- Mobile app is a computerized system subject to validation (IQ/OQ/PQ)
- Electronic signatures may be required for GxP compliance (21 CFR Part 11)

**GDPR:**
- Field technician location during verification may be collected
- Lawful basis: Legitimate interest (asset verification)
- Data minimization: Only collect location necessary for verification

## Validation Rules (Plain Language)

1. **Valid Asset Identifier**: Scanned identifier must match an existing asset (barcode, QR code, or RFID tag).

2. **Technician Authorization**: Only technicians with active verification rights can perform verifications.

3. **Verification Timing**: Verification should occur within the scheduled window (within grace period).

4. **Condition Rating Range**: Rating must be 1, 2, 3, 4, or 5.

5. **GPS Coordinate Validity**: If GPS is captured, latitude must be -90 to 90, longitude -180 to 180.

6. **Photo Size**: Individual photos must not exceed 10MB.

7. **Technician Active Status**: Only active (non-deactivated) technicians can submit verifications.

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| No network during sync | Queue verification locally, retry when online |
| Duplicate scan within 5 minutes | Ignore duplicate |
| Sync conflict detected | Last-write-wins, conflict logged |
| Unknown asset scanned | Create ad-hoc exception record for review |
| Low device storage | Prevent new photos until sync completes |
| Verification overdue beyond grace | Alert manager, still allow verification |

## Verification Data Flow

1. **Scan Asset Identifier** → Validate against asset master
2. **Display Asset Details** → Show expected asset information
3. **Capture Verification Data** → Condition rating, location confirmation, optional photos, GPS, notes
4. **Evaluate Discrepancies** → Compare actual vs. expected
5. **No Discrepancy** → Mark verified, sync to cloud
6. **Discrepancy Found** → Create exception, log, then sync

## Sync States

| State | Description |
|-------|-------------|
| PENDING_SYNC | Verification complete, waiting to sync |
| SYNCED | Successfully synced to cloud |
| CONFLICT | Sync conflict detected, needs resolution |
| EXCEPTION | Discrepancy found, requires review |
