# F-2.2.6 - IoT Sensor Integration

## Requirement ID
F-2.2.6

## Function Name
IoT Sensor Integration

## Function Description
Integrate IoT sensors for asset condition monitoring and predictive maintenance. The system supports multiple sensor types (temperature, vibration, pressure, humidity, current, voltage), configurable data collection frequency, threshold alert settings, and data analysis models for predictive maintenance.

## Business Purpose
Enable real-time monitoring of asset health through IoT sensors to shift from reactive to predictive maintenance. Reduces unplanned downtime, extends asset life, and supports regulatory compliance for safety and environmental monitoring. Provides data-driven insights for maintenance planning and resource allocation.

## Core Business Rules

1. **Sensor Types Supported**: System must support temperature, vibration, pressure, humidity, current, and voltage sensors.

2. **Asset-Sensor Association**: Each sensor must be registered and associated with a specific asset before data collection begins.

3. **Configurable Thresholds**: Each sensor type can have warning and critical threshold settings that trigger alerts when exceeded.

4. **Alert Levels**: 
   - Normal: Readings within acceptable range → Update asset health score
   - Warning: Exceeds warning threshold → Send warning alert
   - Critical: Exceeds critical threshold → Escalate immediately to manager

5. **Sensor Health Monitoring**: If a sensor stops transmitting data beyond the expected interval, generate a sensor offline alert.

6. **Predictive Failure Detection**: Machine learning models analyze sensor history to predict equipment failures within 30 days and generate work order suggestions.

7. **Data Retention**: Raw sensor data retained for 90 days; aggregated data retained for 7 years.

8. **Sensor Calibration**: Readings consistently outside expected range trigger a calibration alert.

9. **Data Quality**: Only process sensor readings marked as "GOOD" quality; uncertain or bad quality data is flagged.

10. **Multiple Sensors**: When multiple sensors of the same type are on one asset, readings are aggregated.

## Data Scope

- **Creates Data:** IoT devices/sensors transmit data; Plant managers configure sensors
- **Views Data:** Same company and plant personnel; cross-plant by company admin
- **Modifies Data:** Plant-level configuration; no user can modify historical sensor data

## Compliance Notes

**OSHA:**
- Industrial equipment monitoring for safety (vibration, temperature)
- 29 CFR 1910.119 (Process Safety Management) requires monitoring for critical equipment
- 29 CFR 1910.147 for equipment lockout/tagout monitoring

**EPA:**
- Environmental emission monitoring under Clean Air Act
- RCRA hazardous waste monitoring for tanks and containers

**GAMP 5 (Pharma):**
- IoT systems are computerized systems subject to validation (IQ/OQ/PQ)
- Electronic records for regulated environments must be retained per 21 CFR Part 11

**GDPR:**
- Sensor data from worker-worn devices may contain personal data
- Lawful basis: Legitimate interest (asset monitoring)
- Data minimization: Only collect sensor types necessary for asset monitoring
- Human oversight required for automated ML-based decisions

## Validation Rules (Plain Language)

1. **Asset IoT-Enabled**: Only assets designated as IoT-enabled can receive sensor data.

2. **Sensor Registered**: Sensor must be registered in the system with an active status.

3. **Valid Association**: Sensor must be associated with the correct asset.

4. **Data Format Valid**: Sensor data must conform to the expected format for that sensor type.

5. **Value Range**: Reading must be within the sensor's measurement range.

6. **Timestamp Valid**: Sensor timestamps must be within 5 minutes of system time.

7. **Data Freshness**: Data older than 24 hours is flagged as stale.

8. **Sensor Active**: Only data from active (non-fault, non-calibration) sensors is processed.

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Sensor goes offline | Generate sensor offline alert when no data received beyond expected interval |
| Missing data points | Interpolate missing values or flag as data gap |
| Multiple sensors same type | Aggregate readings from all sensors |
| Edge computing failure | Fallback to cloud processing with potential latency |
| Calibration drift | Generate calibration alert when readings consistently off |
| Data quality uncertain | Flag for review, do not use for health calculations |
| Predictive model failure predicted | Generate maintenance work order suggestion for review |

## Sensor States

| State | Description |
|-------|-------------|
| REGISTERED | Sensor configured but not yet active |
| ACTIVE | Sensor normal operation, collecting data |
| INACTIVE | Sensor temporarily disabled |
| FAULT | Sensor malfunction detected |
| CALIBRATION | Sensor undergoing calibration |

## Health Score Calculation

Asset health score is calculated as a weighted average of sensor readings, factoring in:
- Vibration factor (30% weight)
- Temperature factor (25% weight)
- Pressure factor (20% weight)
- Humidity factor (15% weight)
- Electrical factor (10% weight)
